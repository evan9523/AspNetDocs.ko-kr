> Visual Studio는 ASP.NET 개발자에 게 두 가지 유형의 프로젝트를 제공합니다. 웹 응용 프로그램 프로젝트 (Wap) 및 웹 사이트 프로젝트 (하며). 두 가지 프로젝트 형식 간의 주요 차이점 중 하나는 Wap 웹 서버의 WSP의 코드를 자동으로 컴파일할 수 있는 반면 배포 하기 전에 명시적으로 컴파일된 코드에 있어야 합니다. 그러나 배포 전에 WSP 미리 컴파일할 수 것입니다. 이 자습서는 미리 컴파일의 이점을 설명 하 고 웹 사이트에서 Visual Studio 내에서 및 명령줄에서 미리 컴파일하는 방법을 보여 줍니다.


## <a name="introduction"></a>소개

Visual Studio는 두 가지 다른 프로젝트 형식 ASP.NET 개발자를 제공합니다. 웹 응용 프로그램 프로젝트 (WAP) 및 웹 사이트 프로젝트 (WSP). 이러한 프로젝트 형식 간의 주요 차이점 중 하나는 Wap 필요 함을 *명시적 컴파일을* 하며 사용 하는 반면 *자동 컴파일을*, 기본적으로 합니다. Wap를 사용 하 여 웹 사이트에서 생성 된 단일 어셈블리로 웹 응용 프로그램의 코드 컴파일할 `Bin` 폴더입니다. 태그 콘텐츠를 복사 하는 배포 시 (합니다 `.aspx.ascx`, 및 `.master` 파일)에서 어셈블리와 함께 프로젝트에는 `Bin` 폴더; 코드 숨김 클래스 파일 자체를 배포할 필요가 없습니다. 반면에 프로덕션 환경으로 태그 페이지와 해당 하는 코드 숨김 클래스를 복사 하 여 하며 배포 합니다. 코드 숨김 클래스를 웹 서버에서 요청 시에 컴파일됩니다.

> [!NOTE]
> 다시 "명시적 컴파일와 자동 컴파일" 섹션을 참조 합니다 [ *결정 파일이 필요한 배포* 자습서](determining-what-files-need-to-be-deployed-cs.md) 프로젝트 간의 차이점에 대 한 자세한 배경 모델, 명시적 및 자동 컴파일 및 컴파일 모델 배포에 미치는 영향


자동 컴파일 옵션은 사용이 간편 합니다. 명시적인 컴파일 단계가 없습니다 및 된 파일만 수정 필요 명시적 컴파일이 변경된 태그 페이지 및 방금 컴파일된 어셈블리를 배포 해야 하는 반면 배포 해야 합니다. 그러나 자동 배포에는 두 가지 잠재적인 단점이 있습니다.

- 먼저 방문할 때 페이지가 자동으로 컴파일해야, 때문에 있을 수 있습니다 짧지만 눈에 띄는 지연이 ASP.NET 페이지는 배포 된 후 처음으로 요청 될 때.
- 자동 컴파일에서는 선언적 태그 및 소스 코드가 웹 서버에 있어야 합니다. 해당 웹 서버에 설치 하는 고객은 웹 응용 프로그램을 판매 하려는 경우이 보기 좋지 않은 옵션 수 있습니다.

경우 단점 위의 두 거래 분리기가, WAP 모델에 두 스위치 수 중 하나 또는 *미리 컴파일할* WSP 배포 하기 전에 합니다. 이 자습서에서는 가장 적합 한 호스팅되는 웹 사이트 미리 컴파일 옵션을 검토 하 고 미리 컴파일 프로세스와 미리 컴파일된 웹 사이트의 배포를 안내 합니다.

## <a name="an-overview-of-aspnet-code-generation-and-compilation"></a>ASP.NET 코드 생성 및 컴파일 개요

사용 가능한 미리 컴파일 옵션을 살펴보기 전에 먼저 살펴보겠습니다 코드 생성 및 컴파일을 작성 또는 마지막으로 업데이트 된 이후 처음으로 ASP.NET 페이지가 요청 될 때 발생 하는 합니다. ASP.NET 페이지 두 부분으로 구성 되 듯이: 선언적 태그를 `.aspx` 파일 및 소스 코드 부분을 별도 코드 숨김 클래스 파일에서 일반적으로 (`.aspx.cs`). ASP.NET 페이지가 요청 될 때 런타임에서 수행 되는 단계는 응용 프로그램의 컴파일 모델에 따라 달라 집니다.

Wap를 사용 하 여 페이지의 소스 코드를 단일 어셈블리로 배포 하기 전에 명시적으로 컴파일된 있어야 합니다. 배포 하는 동안 다양 한 태그 페이지와이 어셈블리는 프로덕션 환경으로 복사 됩니다. 요청을 ASP.NET 페이지에 대 한 웹 서버에 도착 하면 런타임에서 페이지의 코드 숨김 클래스의 인스턴스를 만듭니다 및이 호출 하는 해당 `ProcessRequest` 페이지 수명 주기를 시작 하 고, 궁극적으로 반환 되는 페이지의 콘텐츠를 생성 하는 메서드 요청자에 게 합니다. 코드 숨김 클래스를 이미 배포 전에 어셈블리로 컴파일되므로 런타임 ASP.NET 페이지의 코드 숨김 클래스를 사용 하 여 작업할 수 있습니다.

하며 및 자동 컴파일을 사용 하 여 배포 하기 전에 명시적 컴파일 단계가 없습니다 있습니다. 대신 배포 프로덕션 환경에는 선언적 및 소스 코드 콘텐츠를 복사 해야 합니다. 요청이 도착 하면 ASP.NET 페이지에 대 한 웹 서버에 처음으로 페이지 생성 되거나 마지막으로 업데이트 되므로, 런타임에서 해야 먼저 코드 숨김 클래스를 어셈블리로 컴파일해야 합니다. 이 컴파일된 어셈블리 폴더에 저장 됩니다 `%WINDIR%\Microsoft.NET\Framework\v2.0.50727\Temporary ASP.NET Files`이지만이 폴더의 위치를 통해 사용자 지정할 수 있습니다 합니다 `<compilation tempDirectory="" />` 요소의 `<system.web>`일반적으로, `Web.config`합니다. 어셈블리가 디스크에 저장 하지 않아도 동일한 페이지에 대 한 후속 요청에 다시 컴파일할 수 있습니다.

> [!NOTE]
> 예상한 것 처럼 약간의 지연이 발생 되는 페이지의 코드를 컴파일하고 결과 어셈블리를 저장 하는 서버에 대 한 현재 자동 컴파일을 사용 하는 사이트에 처음으로 (또는 이후 변경 되었음을 처음으로) 페이지를 요청 하는 경우 디스크입니다.


즉, 명시적 컴파일을 사용 하 여 해야 배포 하기 전에 웹 사이트의 소스 코드를 컴파일하려면 해당 단계를 수행 하지 않아도 런타임에 저장 합니다. 자동 컴파일을 사용 하 여 런타임에 만들거나 마지막으로 업데이트 된 후 첫 번째 방문 페이지에 대 한 약간의 초기화 비용은 하지만 페이지의 소스 코드의 컴파일을 처리 합니다.

그러나 ASP.NET 페이지의 선언적 파트에 대 한 (의 `.aspx` 파일)? 관계가 있는지 명확 합니다 `.aspx` 파일과 해당 코드 숨김 클래스에 선언적 태그에 정의 된 웹 컨트롤의 코드는 코드에서 액세스할 수 있습니다. 명백한 것도 하는 콘텐츠는 `.aspx` 파일 페이지에 의해 생성 된 렌더링 된 태그에 크게 영향을 줍니다. 런타임 텍스트, HTML 사용 하 여 작업 및 웹 컨트롤 구문에 정의 된 어떻게는 `.aspx` 요청 된 페이지를 생성 하는 파일의 콘텐츠를 렌더링 된?

Wap 및 하며 다는 구현 세부 정보 sidetracked 너무 가져오기 하지 않으려는 경우 있지만 런타임 보호 된 멤버와 메서드를 다양 한 웹 컨트롤을 포함 하는 클래스 파일을 자동으로 생성 되는 간단히 말해 합니다. 이렇게 생성 된 파일이으로 구현 되는 *partial 클래스* 해당 코드 숨김 클래스에 합니다. ([Partial 클래스](http://www.dotnet-guide.com/partialclasses.html) 단일 클래스의 내용에 대 한 여러 파일에 걸쳐 분산 될 수 있도록 합니다.) 따라서 코드 숨김 클래스는 두 곳에서 정의 됩니다:에 `.aspx.cs` 를 만들고이 자동으로 생성 된 클래스에서 런타임에 의해 생성 된 파일. 이 자동으로 생성 된 클래스에 저장 된 `%WINDIR%\Microsoft.NET\Framework\v2.0.50727\Temporary ASP.NET Files` 폴더입니다.

중요 한 소모할 런타임에서 렌더링 되어야 하는 ASP.NET 페이지에는 다음과 같습니다 선언적 해당 및 소스 코드 부분을 어셈블리로 컴파일해야 합니다. Wap를 사용 하 여 소스 코드에 명시적으로 배포 하기 전에 어셈블리에 컴파일됩니다 있지만 선언적 태그 코드로 변환 및 웹 서버에서 런타임에 의해 컴파일된 계속 해야 합니다. 자동 컴파일을 사용 하며, 소스 코드와 선언적 태그를 모두 웹 서버에서 컴파일되도록 해야 합니다.

WSP 모델을 사용 하 여 명시적 컴파일을 사용 하는 것이 가능 합니다. WAP 모델을 사용 하 여 같은 원본 코드 부분을 명시적으로 컴파일할 수 있습니다. 게다가 선언적 태그를 컴파일할 수 있습니다.

## <a name="precompilation-options"></a>미리 컴파일 옵션

.NET Framework와 함께 제공 되는 [ASP.NET 컴파일 도구 (`aspnet_compiler.exe`)](https://msdn.microsoft.com/library/ms229863.aspx) WSP 모델을 사용 하 여 빌드된 ASP.NET 응용 프로그램의 소스 코드 (및 콘텐츠도)를 컴파일할 수 있도록 합니다. 이 도구는.NET Framework 버전 2.0과 함께 출시 된 및에 `%WINDIR%\Microsoft.NET\Framework\v2.0.50727` 폴더; 명령줄에서 사용 하거나 빌드 메뉴의 웹 사이트 게시 옵션을 통해 Visual Studio 내에서 시작할 수 있습니다.

컴파일 도구에서는 두 가지 일반적인 형태의 컴파일: 원위치 미리 컴파일 및 배포용입니다. 전체 컴파일을 사용 하면 실행할는 `aspnet_compiler.exe` 명령줄 도구를 컴퓨터에 상주 하는 웹 사이트의 실제 경로에 가상 디렉터리의 경로 지정 합니다. 컴파일합니다 각 ASP.NET 페이지의 컴파일된 버전을 저장 합니다. 프로젝트에서 컴파일 도구는 `%WINDIR%\Microsoft.NET\Framework\v2.0.50727\Temporary ASP.NET Files` 경우 페이지가 각 방문한 처음으로 브라우저에서와 마찬가지로 폴더입니다. 전체 미리 컴파일은 런타임을이 단계를 수행할 필요를 완화 하기 때문에 사이트에 새로 배포 된 ASP.NET 페이지에 대 한 첫 번째 요청 속도 높일 수 있습니다. 그러나 명령줄 웹 서버에서 프로그램을 실행할 수 있는지 필요 하기 때문에 전체 미리 컴파일은 대부분 호스팅된 웹 사이트의 유용 하지 않습니다. 공유 호스팅 환경에서이 수준의 액세스 권한 허용 되지 않습니다.

> [!NOTE]
> 전체 미리 컴파일에 대 한 자세한 내용은 체크 아웃 [방법: ASP.NET 웹 사이트를 미리 컴파일할](https://msdn.microsoft.com/library/ms227972.aspx) 하 고 [ASP.NET 2.0의에서 미리 컴파일](http://www.odetocode.com/Articles/417.aspx)합니다.


웹 사이트의 페이지를 컴파일하는 대신는 `Temporary ASP.NET Files` 폴더 배포용 디렉터리를 선택 해 및 프로덕션 환경에 배포할 수 있는 형식으로 페이지를 컴파일합니다.

이 자습서에서 살펴볼 배포용의 두 가지 가지: 업데이트할 수 있는 사용자 인터페이스를 사용 하 여 미리 컴파일 및 업데이트할 수 없는 사용자 인터페이스를 사용 하 여 미리 컴파일입니다. 업데이트할 수 있는 사용자 인터페이스를 사용 하 여 미리 컴파일에 있는 선언적 태그를 유지 합니다 `.aspx`, `.ascx`, 및 `.master` 파일을 확인 하 고, 원하는 경우 수정할 프로덕션 서버의 선언적 태그 개발자 있기 때문입니다. 업데이트할 수 없는 사용자 인터페이스를 사용 하 여 미리 컴파일 생성 `.aspx` void 모든 콘텐츠 및 제거 된 페이지 `.ascx` 및 `.master` 선언적 태그 숨기고 개발자가에서 변경 금지 함으로써 파일을 프로덕션 환경입니다.

### <a name="precompiling-for-deployment-with-an-updatable-user-interface"></a>업데이트할 수 있는 사용자 인터페이스를 사용 하 여 배포를 위한 미리 컴파일

배포용을 이해 하는 가장 좋은 방법은 작업의 예제를 보려는 경우 업데이트할 수 있는 사용자 인터페이스를 사용 하 여 배포에 대 한 책 검토 WSP를 미리 컴파일할 보겠습니다. ASP.NET 컴파일 도구는 Visual Studio의 빌드 메뉴에서 또는 명령줄에서 호출할 수 있습니다. 이 섹션에서는 Visual Studio 내에서 도구를 사용 하 여 검사 명령줄에서 컴파일러 도구를 실행 "the Command Line에서 미리 컴파일" 섹션을 확인 합니다.

마지막으로, 시간을 내어 중 하나를 열려면는 `.aspx` Visual Studio 또는 원하는 텍스트 편집기를 사용 하 여 대상 위치에 있는 파일입니다. 업데이트할 수 있는 사용자 인터페이스를 사용 하 여 배포를 위한 미리 컴파일, 대상 위치 디렉터리의 ASP.NET 페이지를 웹 사이트에서 해당 파일과 정확히 동일한 태그를 포함 합니다.

### <a name="precompiling-from-the-command-line"></a>명령줄에서 미리 컴파일

배후에서 Visual Studio의 웹 사이트 게시 대화 상자 호출 ASP.NET 컴파일 도구 (`aspnet_compiler.exe`) 웹 사이트를 미리 컴파일할 합니다. 또는 명령줄에서이 도구를 호출할 수 있습니다. 사실, Visual Web Developer를 사용 하는 경우 다음 해야 명령줄에서 컴파일러 도구를 실행 하려면 Visual Web Developer의 빌드 메뉴 웹 사이트 게시 옵션을 포함 하지 않습니다.

명령줄에서 컴파일러 도구를 사용 하려면 명령줄에 삭제 framework 디렉터리로 이동 하 여 시작 `%WINDIR%\Microsoft.NET\Framework\v2.0.50727`합니다. 명령줄에 다음 문을 입력 어.

`aspnet_compiler -p "physical_path_to_app" -v / -f -u "target_location_folder"`

위의 명령은 ASP.NET 컴파일러 도구 시작 (`aspnet_compiler.exe`)를 통해 합니다 `-p` 스위치를 기반으로 웹 사이트 미리 컴파일 하도록 지시 *물리적\_경로\_에\_앱*; 와 같이이 값은 `C:\MySites\BookReviews`, 인용 부호 구분 해야 하 고 있습니다.

`-v` 스위치 사이트의 가상 디렉터리를 지정 합니다. 사이트는 기본 웹 사이트의 IIS 메타 베이스로 등록 된 경우 생략할 수는 `-p` 전환 하 고 방금 응용 프로그램의 가상 디렉터리를 지정 합니다. 사용 하는 경우는 `-p` 스위치를 값 계속을 `-v` 스위치 웹 사이트의 루트를 나타내며 응용 프로그램 루트 참조를 확인 하는 데 사용 됩니다. 예를 들어 값을 지정 하는 경우 `-v /MySite` 응용 프로그램에 다음 참조 `~/path/file` 로 확인 될 `~/MySite/path/file`합니다. 스위치를 사용 했습니다도 서 리뷰 사이트는 웹 호스팅 회사 내에서 루트 디렉터리에 있으므로 `-v /`합니다.

`-f` 스위치에 있는 경우 지시를 덮어쓰려면 컴파일 도구는 *대상\_위치\_폴더* 이미 존재 하는 경우 디렉터리입니다. 생략 하면는 `-f` 스위치와 대상 위치 폴더가 이미 있으면 컴파일 도구 오류와 함께 종료 됩니다. "오류 ASPRUNTIME: 대상 디렉터리가 비어 있지 않습니다. 하십시오. 수동으로 삭제 하거나 다른 대상을 선택 합니다. "

`-u` 스위치 있으면 알립니다는 업데이트할 수 있는 사용자 인터페이스를 만드는 도구입니다. 이 스위치를 업데이트할 수 없는 사용자 인터페이스를 사용 하 여 사이트를 미리 컴파일할를 생략 합니다.

마지막으로 *대상\_위치\_폴더* 대상 위치 디렉터리의 실제 경로 같이이 값은 `C:\MySites\Output\BookReviews`, 따옴표로 구분 해야 하 고 있습니다.

## <a name="deploying-the-precompiled-website"></a>미리 컴파일된 웹 사이트 배포

이 시점에서 업데이트 가능 하 고 업데이트할 수 없는 사용자 인터페이스 옵션 모두를 사용 하 여 웹 사이트를 미리 컴파일할 ASP.NET 컴파일 도구를 사용 하는 방법을 살펴보았습니다. 그러나 예제 지금까지 미리 컴파일한 웹 사이트를 로컬 폴더 및 프로덕션 환경에 없습니다. 좋은 소식은 간단히 미리 컴파일된 웹 사이트 배포를 수행할 수 있습니다 Visual Studio 또는 기타 파일 복사 메커니즘을 통해 같은 독립 실행형 FTP 클라이언트에서 보여 줍니다.

웹 사이트 게시 대화 상자 (먼저 나오는 **그림 1**)에 미리 컴파일된 웹 사이트 파일을 복사할 위치를 나타내는 대상 위치 옵션을 합니다. 이 위치는 원격 웹 서버 또는 FTP 서버를 수 있습니다. 이 텍스트 상자에 원격 서버를 입력 미리 컴파일한 웹 사이트를 한 번에 지정된 된 서버에 배포 합니다. 또는 로컬 폴더에 웹 사이트를 미리 컴파일할 수 있으며 다음 FTP 또는 다른 접근 방식을 통해 프로덕션 환경으로 해당 폴더의 콘텐츠 수동으로 복사 수 있습니다.

Visual Studio의 웹 사이트 게시 대화 상자를 통해 자동으로 배포 하는 미리 컴파일된 웹 사이트는 것이 간단한 사이트에 대 한 유용한 경우 개발 및 프로덕션 환경 간의 구성 차이가 있습니다. 그러나 설명한 것 처럼 합니다 [ *일반적인 구성 차이 간의 개발 및 프로덕션* 자습서](common-configuration-differences-between-development-and-production-cs.md) 존재 이러한 차이점에 대해서도 드물지 않습니다. 예를 들어,도 서 리뷰 웹 응용 프로그램 개발 환경에서 보다 프로덕션 환경에서 다른 데이터베이스를 사용합니다. Visual Studio에서 원격 서버에 웹 사이트를 게시할 때 무조건 개발 환경에서 구성 파일 정보를 복사 합니다.

개발 및 프로덕션 환경 간의 구성 차이 사용 하 여 사이트에 대 한 로컬 디렉터리로 사이트 미리 컴파일, 프로덕션 환경 특정 구성 파일을 복사 및 다음 미리 컴파일된 출력의 내용을 복사 하는 최선의 수 있습니다. 프로덕션입니다.

프로덕션 환경으로 개발 환경에서 파일을 복사 하는 대 한 참조를 [ *FTP 클라이언트를 사용 하 여 웹 사이트 배포* ](deploying-your-site-using-an-ftp-client-cs.md) 하 고 [  *웹 사이트를 사용 하 여 Visual Studio를 배포* ](determining-what-files-need-to-be-deployed-cs.md) 자습서입니다.

## <a name="summary"></a>요약

ASP.NET 컴파일 두 모드를 지원 합니다: 자동 및 명시적입니다. 이전 자습서에서 설명 했 듯이 웹 응용 프로그램 프로젝트 (Wap) 웹 사이트 프로젝트 (하며) 기본적으로 자동 컴파일을 사용 하는 반면 명시적 컴파일을 사용 합니다. 그러나 명시적으로 배포 하기 전에 WSP ASP.NET 컴파일 도구를 사용 하 여 컴파일할 수 것입니다.

이 자습서 배포 지원에 대 한 컴파일 도구의 미리 컴파일 집중합니다. 배포를 위한 미리 컴파일, 하는 경우 컴파일 도구 대상 위치 폴더를 만듭니다, 지정 된 웹 응용 프로그램의 소스 코드를 컴파일하고 이러한 복사 대상 위치 폴더 어셈블리와 콘텐츠 파일이 컴파일됩니다. 업데이트 가능 또는 불가능 사용자 인터페이스를 만들려면 컴파일 도구를 구성할 수 있습니다. 업데이트할 수 없는 사용자 인터페이스 옵션을 사용한 미리 컴파일, 콘텐츠 파일에 있는 선언적 태그 제거 됩니다. 간단히 말해 미리 컴파일 허용 원하는 경우 웹 사이트 프로젝트 기반 응용 프로그램을 제거 하는 선언적 태그와 모든 소스 코드 파일을 포함 하지 않고 배포할 수 있습니다.
