# <a name="profiles-themes-and-web-parts"></a>프로필, 테마 및 웹 파트

[로 마이크로 소프트](https://github.com/microsoft)

> ASP.NET 2.0의 구성 및 계측에는 큰 변화가 있습니다. 새로운 ASP.NET 구성 API를 사용하면 프로그래밍 방식으로 구성을 변경할 수 있습니다. 또한 많은 새로운 구성 설정이 존재하여 새로운 구성및 계측이 가능합니다.

## <a name="profile-providers"></a>프로필 공급자

ASP.NET 프로파일은 공급자 모델을 사용합니다. 기본 공급자는 SqlProfileProvider 공급자를 사용하여 웹\_응용 프로그램의 앱 데이터 폴더에 SQL Server Express 데이터베이스에 정보를 저장합니다. 데이터베이스가 없으면 프로필이 정보를 저장하려고 할 때 ASP.NET 자동으로 데이터베이스가 생성됩니다.

그러나 경우에 따라 사용자 고유의 프로필 공급자를 개발할 수 있습니다. ASP.NET 프로파일 기능을 사용하면 다른 공급자를 쉽게 사용할 수 있습니다.

다음과 같은 경우 사용자 지정 프로필 공급자를 만듭니다.

- .NET Framework에 포함된 프로필 공급자가 지원하지 않는 FoxPro 데이터베이스 또는 Oracle 데이터베이스와 같은 데이터 원본에 프로필 정보를 저장해야 합니다.
- .NET Framework에 포함된 공급자가 사용하는 데이터베이스 스키마와 다른 데이터베이스 스키마를 사용하여 프로필 정보를 관리해야 합니다. 일반적인 예는 프로필 정보를 기존 SQL Server 데이터베이스의 사용자 데이터와 통합하려는 것입니다.

### <a name="required-classes"></a>필수 수업

프로필 공급자를 구현하려면 System.Web.Profile.Profile.ProfileProvider 추상 클래스를 상속하는 클래스를 만듭니다. **ProfileProvider** 추상 클래스는 System.Configuration.Settings.Settings공급자 추상 클래스를 상속하며, 이 클래스는 System.Configuration.Provider.Provider.ProviderBase 추상 클래스를 상속합니다. 이 상속 체인 때문에 **ProfileProvider** 클래스의 필수 멤버 외에도 **SettingsProvider** 및 **ProviderBase** 클래스의 필수 멤버를 구현해야 합니다.

다음 표에서는 **ProviderBase,** **SettingsProvider**및 **ProfileProvider** 추상 클래스에서 구현해야 하는 속성 및 메서드를 설명합니다.

## <a name="aspnet-themes"></a>ASP.NET 테마

## <a name="what-are-aspnet-20-themes"></a>ASP.NET 2.0 테마란?

웹 응용 프로그램의 가장 중요한 측면 중 하나는 사이트 전체에서 일관된 모양과 느낌입니다. ASP.NET 1.x 개발자는 일반적으로 CSS(계단식 스타일 시트)를 사용하여 일관된 모양과 느낌을 구현합니다. ASP.NET 2.0 테마는 ASP.NET 개발자에게 html 요소뿐만 아니라 ASP.NET 서버 컨트롤의 모양을 정의할 수 있기 때문에 CSS에서 크게 개선됩니다. ASP.NET 테마는 개별 컨트롤, 특정 웹 페이지 또는 전체 웹 응용 프로그램에 적용할 수 있습니다. 테마는 CSS 파일, 선택적 스킨 파일 및 이미지가 필요한 경우 선택적 이미지 디렉토리의 조합을 사용합니다. 스킨 파일은 서버 컨트롤의 시각적 모양을 ASP.NET 제어합니다.

## <a name="where-are-themes-stored"></a>테마는 어디에 저장되어 있습니까?

테마가 저장되는 위치는 범위에 따라 다릅니다. 모든 응용 프로그램에 적용할 수 있는 테마는 다음 폴더에 저장됩니다.

`C:\WINDOWS\Microsoft.NET\Framework\v2.x.xxxxx\ASP.NETClientFiles\Themes\<Theme_Name>`

특정 응용 프로그램과 관련된 테마는 웹 사이트의 `App\_Themes\<Theme\_Name>` 루트에 있는 디렉터리에 저장됩니다.

> [!NOTE]
> 스킨 파일은 모양에 영향을 주는 서버 제어 속성만 수정해야 합니다.

전역 테마는 웹 서버에서 실행되는 모든 응용 프로그램 이나 웹 사이트에 적용할 수 있는 테마입니다. 이러한 테마는 기본적으로 v2.x.xxxxx 디렉터리 안에 있는 ASP.NETClientfiles\테마 디렉토리에 저장됩니다. 또는 웹 사이트의 루트에 있는 aspnet\_클라이언트/시스템\_웹/[버전]/테마/[테마\_이름] 폴더로 테마 파일을 이동할 수 있습니다.

응용 프로그램별 테마는 파일이 있는 응용 프로그램에만 적용할 수 있습니다. 이러한 파일은 웹 `App\_Themes/<theme\_name>` 사이트의 루트에 있는 디렉터리에 저장됩니다.

## <a name="web-parts"></a>웹 파트

ASP.NET 웹 파트는 최종 사용자가 브라우저에서 직접 웹 페이지의 콘텐츠, 모양 및 동작을 수정할 수 있도록 웹 사이트를 만들기 위한 통합된 컨트롤 집합입니다. 수정 사항은 사이트의 모든 사용자 또는 개별 사용자에게 적용할 수 있습니다. 사용자가 페이지와 컨트롤을 수정하면 설정을 저장하여 향후 브라우저 세션에서 사용자의 개인 기본 설정을 유지할 수 있습니다. 이러한 웹 파트 기능은 개발자가 개발자 나 관리자의 개입 없이 최종 사용자가 웹 응용 프로그램을 동적으로 개인화할 수 있도록 권한을 부여할 수 있음을 의미합니다.

웹 부품 컨트롤 세트를 사용하면 개발자가 최종 사용자가 다음을 수행할 수 있습니다.

- 페이지 콘텐츠를 개인 설정합니다. 사용자는 페이지에 새 웹 파트 컨트롤을 추가하거나, 제거하거나, 숨기거나, 일반 창처럼 최소화할 수 있습니다.
- 페이지 레이아웃을 개인화합니다. 사용자는 웹 파트 컨트롤을 페이지의 다른 영역으로 끌거나 모양, 속성 및 동작을 변경할 수 있습니다.
- 컨트롤내보내기 및 가져오기 사용자는 다른 페이지 나 사이트에서 사용할 수 있는 웹 파트 컨트롤 설정을 가져오거나 내보낼 수 있으며 컨트롤의 속성, 모양 및 데이터를 유지할 수 있습니다. 이렇게 하면 최종 사용자의 데이터 입력 및 구성 요구가 줄어듭니다.
- 연결을 만듭니다. 예를 들어 차트 컨트롤이 주식 시세 표시기 컨트롤의 데이터에 대한 그래프를 표시할 수 있도록 컨트롤 간의 연결을 설정할 수 있습니다. 사용자는 연결 자체뿐만 아니라 차트 컨트롤이 데이터를 표시하는 방법의 모양과 세부 사항을 개인화 할 수 있습니다.
- 사이트 수준 설정을 관리하고 개인화합니다. 권한이 있는 사용자는 사이트 수준 설정을 구성하고, 사이트 또는 페이지에 액세스할 수 있는 사용자를 결정하고, 컨트롤에 대한 역할 기반 액세스를 설정할 수 있습니다. 예를 들어 관리 역할의 사용자는 모든 사용자가 웹 파트 컨트롤을 공유하도록 설정하고 관리자가 아닌 사용자가 공유 컨트롤을 개인 설정하지 못하도록 할 수 있습니다.

일반적으로 웹 파트 컨트롤을 사용하는 페이지 만들기, 개별 웹 파트 컨트롤 만들기 또는 포털과 같은 완전하고 개인화 가능한 웹 응용 프로그램 만들기의 세 가지 방법 중 하나로 웹 파트를 사용합니다.

## <a name="page-development"></a>페이지 개발

페이지 개발자는 Microsoft Visual Studio 2005와 같은 시각적 디자인 도구를 사용하여 웹 파트를 사용하는 페이지를 만들 수 있습니다. Visual Studio와 같은 도구를 사용할 때의 한 가지 장점은 웹 파트 컨트롤 세트가 시각적 디자이너에서 웹 파트 컨트롤의 끌어서 놓기 생성 및 구성을 위한 기능을 제공한다는 것입니다. 예를 들어 디자이너를 사용하여 웹 파트 영역 또는 웹 파트 편집기 컨트롤을 디자인 표면으로 드래그한 다음 웹 파트 컨트롤 세트에서 제공하는 UI를 사용하여 디자이너에서 컨트롤을 올바르게 구성할 수 있습니다. 이렇게 하면 웹 파트 응용 프로그램의 개발 속도가 빨라지고 작성해야 하는 코드의 양이 줄어듭니다.

## <a name="control-development"></a>제어 개발

표준 웹 서버 컨트롤, 사용자 지정 서버 컨트롤 및 사용자 컨트롤을 비롯한 기존 ASP.NET 컨트롤을 웹 부품 컨트롤로 사용할 수 있습니다. 환경을 프로그래밍 방식으로 최대한 제어하려면 WebPart 클래스에서 파생된 사용자 지정 웹 파트 컨트롤을 만들 수도 있습니다. 개별 웹 부품 컨트롤 개발의 경우 일반적으로 사용자 컨트롤을 만들고 웹 부품 컨트롤로 사용하거나 사용자 지정 웹 부품 컨트롤을 개발합니다.

사용자 지정 웹 파트 컨트롤을 개발하는 예로, 캘린더, 목록, 금융 정보, 뉴스, 계산기, 콘텐츠 업데이트를 위한 리치 텍스트 컨트롤, 데이터베이스에 연결하는 편집 가능한 그리드, 디스플레이를 동적으로 업데이트하는 차트 등 다른 ASP.NET 서버 컨트롤에서 제공하는 기능을 제공하는 컨트롤을 만들 수 있습니다. 또는 날씨 및 여행 정보. 시각적 디자이너에게 컨트롤을 제공하는 경우 Visual Studio를 사용하는 모든 페이지 개발자는 추가 코드를 작성하지 않고도 컨트롤을 웹 파트 영역으로 끌어와 디자인 타임에 구성할 수 있습니다.

개인 화는 웹 부품 기능의 기초입니다. 이를 통해 사용자는 페이지에서 웹 파트 컨트롤의 레이아웃, 모양 및 동작을 수정하거나 개인 설정할 수 있습니다. 개인 설정된 설정은 오래 지속됩니다: 현재 브라우저 세션(보기 상태와 마찬가지로)뿐만 아니라 장기 저장소에서도 유지되므로 사용자의 설정도 향후 브라우저 세션에 저장됩니다. 웹 파트 페이지에 대해 개인 설정이 기본적으로 활성화됩니다.

UI 구조 구성 요소는 개인 설정에 의존하며 모든 웹 파트 컨트롤에 필요한 핵심 구조와 서비스를 제공합니다. 모든 웹 파트 페이지에 필요한 UI 구조 구성 요소 중 하나는 WebPartManager 컨트롤입니다. 보이지 는 않지만 이 컨트롤에는 페이지의 모든 웹 파트 컨트롤을 조정하는 중요한 작업이 있습니다. 예를 들어 모든 개별 웹 파트 컨트롤을 추적합니다. 웹 파트 영역(페이지의 웹 파트 컨트롤이 포함된 영역)과 어떤 컨트롤이 있는 영역을 관리합니다. 또한 찾아보기, 연결, 편집 또는 카탈로그 모드와 같이 페이지가 있을 수 있는 다양한 디스플레이 모드를 추적하고 제어하며, 개인 설정 변경 사항이 모든 사용자 또는 개별 사용자에게 적용되는지 여부도 추적합니다. 마지막으로 웹 파트 컨트롤 간의 연결 및 통신을 시작하고 추적합니다.

UI 구조 구성 요소의 두 번째 종류는 영역입니다. 영역은 웹 파트 페이지에서 레이아웃 관리자역할을 합니다. 부품 클래스(부품 컨트롤)에서 파생되는 컨트롤을 포함하고 구성하며, 가로 또는 세로 방향으로 모듈식 페이지 레이아웃을 수행할 수 있는 기능을 제공합니다. 또한 영역은 포함된 각 컨트롤에 대해 공통적이고 일관된 UI 요소(예: 헤더 및 바닥글 스타일, 제목, 테두리 스타일, 작업 단추 등)를 제공합니다. 이러한 공통 요소를 컨트롤의 크롬이라고 합니다. 여러 특수 유형의 영역이 다양한 디스플레이 모드와 다양한 컨트롤에서 사용됩니다. 다양한 유형의 영역은 아래의 웹 파트 필수 제어 섹션에 설명되어 있습니다.

**부품** 클래스에서 파생되는 웹 파트 UI 컨트롤은 웹 파트 페이지의 기본 UI로 구성됩니다. 웹 부품 컨트롤 세트는 부품 컨트롤을 작성할 때 제공하는 옵션에 유연하고 포괄적입니다. 사용자 지정 웹 파트 컨트롤을 만드는 것 외에도 기존 ASP.NET 서버 컨트롤, 사용자 컨트롤 또는 사용자 지정 서버 컨트롤을 웹 파트 컨트롤로 사용할 수도 있습니다. 웹 파트 페이지를 만드는 데 가장 일반적으로 사용되는 필수 컨트롤은 다음 섹션에 설명되어 있습니다.

## <a name="web-parts-essential-controls"></a>웹 파트 필수 컨트롤

웹 파트 컨트롤 집합은 광범위하지만 일부 컨트롤은 웹 파트가 작동하는 데 필요하거나 웹 파트 페이지에서 가장 자주 사용되는 컨트롤이기 때문에 필수적입니다. 웹 파트를 사용하고 기본 웹 파트 페이지를 만들기 시작할 때 다음 표에 설명된 필수 웹 파트 컨트롤을 숙지하는 것이 좋습니다.

| **웹 파트 컨트롤** | **설명** |
| --- | --- |
| Webpartmanager | 페이지의 모든 웹 파트 컨트롤을 관리합니다. 모든 웹 파트 페이지에 대해 하나만 **웹파트Manager** 컨트롤이 필요합니다. |
| Catalogzone | 카탈로그파트 컨트롤이 포함되어 있습니다. 이 영역을 사용하여 사용자가 페이지에 추가할 컨트롤을 선택할 수 있는 웹 파트 컨트롤 카탈로그를 만듭니다. |
| Editorzone | 편집기 파트 컨트롤을 포함합니다. 이 영역을 사용하면 사용자가 페이지에서 웹 파트 컨트롤을 편집하고 개인 설정할 수 있습니다. |
| Webpartzone | 페이지의 기본 UI를 구성하는 WebPart 컨트롤에 대한 전체 레이아웃을 포함하고 제공합니다. 웹 파트 컨트롤이 있는 페이지를 만들 때마다 이 영역을 사용합니다. 페이지에는 하나 이상의 영역이 포함될 수 있습니다. |
| Connectionszone | WebPartConnection 컨트롤을 포함 하 고 연결을 관리 하기 위한 UI를 제공 합니다. |
| 웹 파트(제네릭웹파트) | 기본 UI를 렌더링합니다. 대부분의 웹 파트 UI 컨트롤은 이 범주에 속합니다. 프로그래밍 방식의 최대 제어를 위해 기본 **WebPart** 컨트롤에서 파생된 사용자 지정 웹 파트 컨트롤을 만들 수 있습니다. 기존 서버 컨트롤, 사용자 컨트롤 또는 사용자 지정 컨트롤을 웹 파트 컨트롤로 사용할 수도 있습니다. 이러한 컨트롤이 영역에 배치될 때마다 **WebPartManager** 컨트롤은 런타임에 **GenericWebPart** 컨트롤로 자동으로 래핑하여 웹 파트 기능과 함께 사용할 수 있도록 합니다. |
| Catalogpart | 사용자가 페이지에 추가할 수 있는 사용 가능한 웹 파트 컨트롤 목록이 포함되어 있습니다. |
| 웹 파트 커넥션 | 페이지에서 두 웹 파트 컨트롤 간의 연결을 만듭니다. 연결은 웹 파트 컨트롤 중 하나를 공급자(데이터)로 정의하고 다른 하나는 소비자로 정의합니다. |
| Editorpart | 특수 편집기 컨트롤의 기본 클래스역할을 합니다. |
| 에디터 파트 컨트롤(외관편집파트, 레이아웃편집파트, 동작편집파트, 프로퍼티그리드에디터파트) | 사용자가 페이지에서 웹 파트 UI 컨트롤의 다양한 측면을 개인화할 수 있도록 허용 |

## <a name="lab-create-a-web-part-page"></a>랩: 웹 파트 페이지 만들기

이 실습에서는 ASP.NET 프로필을 통해 정보를 유지하는 웹 파트 페이지를 만듭니다.

### <a name="creating-a-simple-page-with-web-parts"></a>웹 파트를 통해 간단한 페이지 만들기

이 연습부분에서는 웹 파트 컨트롤을 사용하여 정적 콘텐츠를 표시하는 페이지를 만듭니다. 웹 파트 작업의 첫 번째 단계는 두 가지 필수 구조 요소가 있는 페이지를 만드는 것입니다. 먼저 웹 파트 페이지에는 모든 웹 파트 컨트롤을 추적하고 조정하기 위해 WebPartManager 컨트롤이 필요합니다. 둘째, 웹 파트 페이지에는 WebPart 또는 다른 서버 컨트롤을 포함하고 페이지의 지정된 영역을 차지하는 복합 컨트롤인 하나 이상의 영역이 필요합니다.

> [!NOTE]
> 웹 파트 개인 화를 활성화하기 위해 아무 것도 할 필요가 없습니다. 웹 파트 컨트롤 집합에 대해 기본적으로 활성화됩니다. 사이트에서 웹 파트 페이지를 처음 실행할 때 ASP.NET 기본 개인 설정 공급자를 설정하여 사용자 개인 설정 설정을 저장합니다. 개인 지정에 대한 자세한 내용은 웹 파트 개인 정보 조회를 참조하십시오.

### <a name="to-create-a-page-for-containing-web-parts-controls"></a>웹 부품 컨트롤을 포함하는 페이지를 만들려면

1. 기본 페이지를 닫고 WebPartsDemo.aspx라는 사이트에 새 페이지를 추가합니다.
2. **설계** 보기로 전환합니다.
3. **보기** 메뉴에서 UI가 없는 레이아웃 태그 및 컨트롤을 볼 수 있도록 **비시각적 컨트롤** 및 **세부 정보** 옵션이 선택되어 있는지 확인합니다.
4. 디자인 표면의 `<div>` 태그 앞에 삽입 점을 배치하고 ENTER를 눌러 새 선을 추가합니다. 새 줄 문자 앞에 삽입 점을 배치하고 메뉴에서 **블록 형식** 드롭다운 목록 컨트롤을 클릭하고 **제목 1** 옵션을 선택합니다. 제목에 웹 파트 **데모 페이지를**추가합니다.
5. 도구 상자의 **WebParts** 탭에서 **WebPartManager** 컨트롤을 페이지로 드래그하여 새 줄 문자 바로 뒤와 `<div>`태그 앞에 배치합니다.   
  
   **WebPartManager** 컨트롤은 출력을 렌더링하지 않으므로 디자이너 표면에 회색 상자로 나타납니다.
6. `<div>` 삽입 점을 태그 내에 배치합니다.
7. **레이아웃** 메뉴에서 **테이블 삽입을**클릭하고 행 하나와 세 개의 열이 있는 새 테이블을 만듭니다. 셀 **속성** 단추를 클릭하고 **수직 정렬** 드롭다운 목록에서 **상단을** 선택하고 **확인을**클릭한 다음 **확인을** 다시 클릭하여 테이블을 만듭니다.
8. WebPartZone 컨트롤을 왼쪽 테이블 열로 끕습니다. **WebPartZone** 컨트롤을 마우스 오른쪽 단추로 클릭하고 **속성을**선택하고 다음 속성을 설정합니다.   
  
   ID: 사이드바존   
  
   헤더텍스트: 사이드바
9. 두 번째 **WebPartZone** 컨트롤을 가운데 테이블 열로 끌어와 다음 속성을 설정합니다.   
  
   ID: 메인존   
  
   헤더텍스트: 메인
10. 파일을 저장합니다.

이제 페이지에 별도로 제어할 수 있는 두 개의 영역이 있습니다. 그러나 두 영역에는 콘텐츠가 없으므로 콘텐츠를 만드는 것이 다음 단계입니다. 이 연습에서는 정적 콘텐츠만 표시하는 웹 파트 컨트롤로 작업합니다.

### <a name="enabling-users-to-edit-pages-and-change-layout"></a>사용자가 페이지를 편집하고 레이아웃을 변경할 수 있도록 설정

Web Parts는 사용자가 한 영역에서 다른 영역으로 드래그하여 웹 파트 컨트롤의 레이아웃을 변경할 수 있는 기능을 제공합니다. 사용자가 **WebPart** 컨트롤을 한 영역에서 다른 영역으로 이동할 수 있도록 허용하는 것 외에도 사용자가 모양, 레이아웃 및 동작을 포함하여 컨트롤의 다양한 특성을 편집할 수 있도록 허용할 수 있습니다. 웹 파트 컨트롤 세트는 **WebPart** 컨트롤에 대한 기본 편집 기능을 제공합니다. 이 연습에서는 이 연습에서는 그렇게 하지 않지만 사용자가 **WebPart** 컨트롤의 기능을 편집할 수 있도록 사용자 지정 편집기 컨트롤을 만들 수도 있습니다. **WebPart** 컨트롤의 위치를 변경하는 것과 마찬가지로 컨트롤의 속성을 편집하는 것은 사용자가 변경한 내용을 저장하기 위해 ASP.NET 개인 지정에 의존합니다.

이 연습의 이 부분에서는 사용자가 페이지의 **WebPart** 컨트롤의 기본 특성을 편집할 수 있는 기능을 추가합니다. 이러한 기능을 사용하려면 asp:editorzone &lt;&gt; 요소와 두 개의 편집 컨트롤과 함께 페이지에 다른 사용자 지정 사용자 컨트롤을 추가합니다.

### <a name="adding-web-parts-at-run-time"></a>런타임에 웹 파트 추가

사용자가 런타임에 페이지에 웹 파트 컨트롤을 추가하도록 허용할 수도 있습니다. 이렇게 하려면 사용자가 사용할 수 있도록 하려는 웹 부품 컨트롤 목록이 포함된 웹 부품 카탈로그로 페이지를 구성합니다.

**사용자가 런타임에 웹 파트를 추가할 수 있도록 하려면**

1. WebPartsDemo.aspx 페이지를 열고 **디자인** 보기로 전환합니다.
2. 도구 상자의 **WebParts** 탭에서 CatalogZone 컨트롤을 편집기 Zone 컨트롤 아래 테이블의 오른쪽 열로 **드래그합니다.**   
  
   두 컨트롤은 동시에 표시되지 않으므로 동일한 테이블 셀에 있을 수 있습니다.
3. 속성 창에서 **CatalogZone** 컨트롤의 HeaderText 속성에 **웹 파트 추가** 문자열을 할당합니다.

6. 도구 상자의 **표준** 섹션에서 **FileUpload** 컨트롤 및 **캘린더** 컨트롤을 **선언카탈로그파트** 컨트롤의 **WebPartsTemplate** 섹션으로 드래그합니다.
