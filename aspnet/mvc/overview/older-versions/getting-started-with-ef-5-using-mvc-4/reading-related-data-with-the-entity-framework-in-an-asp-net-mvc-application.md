## <a name="lazy-eager-and-explicit-loading-of-related-data"></a><span data-ttu-id="940c6-116">관련된 데이터의 지연 로딩과 명시적 로드</span><span class="sxs-lookup"><span data-stu-id="940c6-116">Lazy, Eager, and Explicit Loading of Related Data</span></span>

<span data-ttu-id="940c6-117">Entity Framework 관련된 데이터를 엔터티의 탐색 속성으로 로드할 수는 여러 가지가 있습니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-117">There are several ways that the Entity Framework can load related data into the navigation properties of an entity:</span></span>

- <span data-ttu-id="940c6-123">*즉시 로드*.</span><span class="sxs-lookup"><span data-stu-id="940c6-123">*Eager loading*.</span></span> <span data-ttu-id="940c6-124">엔터티를 읽을 때 관련된 데이터가 함께 검색됩니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-124">When the entity is read, related data is retrieved along with it.</span></span> <span data-ttu-id="940c6-125">이는 일반적으로 필요한 데이터를 모두 검색하는 단일 조인 쿼리를 발생시킵니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-125">This typically results in a single join query that retrieves all of the data that's needed.</span></span> <span data-ttu-id="940c6-126">즉시 로드를 사용 하 여 지정 된 `Include` 메서드.</span><span class="sxs-lookup"><span data-stu-id="940c6-126">You specify eager loading by using the `Include` method.</span></span>

- <span data-ttu-id="940c6-128">*명시적 로드*.</span><span class="sxs-lookup"><span data-stu-id="940c6-128">*Explicit loading*.</span></span> <span data-ttu-id="940c6-129">이 지연 로드를 제외 하 코드에서 관련된 데이터를 명시적으로 검색 탐색 속성에 액세스할 때 자동으로 발생 하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-129">This is similar to lazy loading, except that you explicitly retrieve the related data in code; it doesn't happen automatically when you access a navigation property.</span></span> <span data-ttu-id="940c6-130">관련된 데이터를 수동으로 로드 하는 엔터티 및 호출에 대 한 개체 상태 관리자 항목을 가져와서 합니다 `Collection.Load` 컬렉션에 대 한 메서드 또는 `Reference.Load` 단일 엔터티를 포함 하는 속성에 대 한 메서드.</span><span class="sxs-lookup"><span data-stu-id="940c6-130">You load related data manually by getting the object state manager entry for an entity and calling the `Collection.Load` method for collections or the `Reference.Load` method for properties that hold a single entity.</span></span> <span data-ttu-id="940c6-131">(다음 예제에서는 관리자 탐색 속성을 로드 하려는 경우 바꿔서 `Collection(x => x.Courses)` 사용 하 여 `Reference(x => x.Administrator)`.)</span><span class="sxs-lookup"><span data-stu-id="940c6-131">(In the following example, if you wanted to load the Administrator navigation property, you'd replace `Collection(x => x.Courses)` with `Reference(x => x.Administrator)`.)</span></span>

<span data-ttu-id="940c6-133">속성 값을 즉시 검색 하지 않으므로 지연 로드 및 명시적 로딩은 둘 다 라고도 *지연 된 로드*합니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-133">Because they don't immediately retrieve the property values, lazy loading and explicit loading are also both known as *deferred loading*.</span></span>

<span data-ttu-id="940c6-134">일반적으로 알고 있는 경우 해야 관련된 데이터 검색을 즉시 로드는 최상의 성능을 제공 하는 모든 엔터티에 대 한 데이터베이스에 전송 된 단일 쿼리 검색 된 각 엔터티에 대 한 별도 쿼리보다 일반적으로 더 효율적 이므로 합니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-134">In general, if you know you need related data for every entity retrieved, eager loading offers the best performance, because a single query sent to the database is typically more efficient than separate queries for each entity retrieved.</span></span> <span data-ttu-id="940c6-135">예를 들어, 위의 예에서 각 부서에 10 개의 관련 된 강좌가 가정 합니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-135">For example, in the above examples, suppose that each department has ten related courses.</span></span> <span data-ttu-id="940c6-136">즉시 로드 예제는 데이터베이스에 왕복만 단일 (조인) 쿼리 및 단일으로 반환 됩니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-136">The eager loading example would result in just a single (join) query and a single round trip to the database.</span></span> <span data-ttu-id="940c6-137">지연 로드 하 고 명시적 로드 예제는 둘 다 11 개의 쿼리 및 11 개 왕복으로 인해 데이터베이스.</span><span class="sxs-lookup"><span data-stu-id="940c6-137">The lazy loading and explicit loading examples would both result in eleven queries and eleven round trips to the database.</span></span> <span data-ttu-id="940c6-138">데이터베이스에 대한 추가 왕복은 대기 시간이 길 때 성능에 특히 악영향을 줍니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-138">The extra round trips to the database are especially detrimental to performance when latency is high.</span></span>

<span data-ttu-id="940c6-139">반면에 일부 시나리오에서는 지연 로딩을 더 효율적입니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-139">On the other hand, in some scenarios lazy loading is more efficient.</span></span> <span data-ttu-id="940c6-140">즉시 로드는 SQL Server를 효율적으로 처리할 수 없습니다는 매우 복잡 한 조인이 생성 될 발생할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-140">Eager loading might cause a very complex join to be generated, which SQL Server can't process efficiently.</span></span> <span data-ttu-id="940c6-141">또는 처리 하는 엔터티 집합의 하위 집합에 대해서만 엔터티의 탐색 속성에 액세스 해야 할 경우, 지연 로드는 즉시 로드는 필요한 것 보다 더 많은 데이터를 검색 하므로 더 잘 수행할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-141">Or if you need to access an entity's navigation properties only for a subset of a set of entities you're processing, lazy loading might perform better because eager loading would retrieve more data than you need.</span></span> <span data-ttu-id="940c6-142">성능이 중요한 경우 최상의 선택을 위해 두 가지 방식으로 성능을 테스트하는 것이 가장 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-142">If performance is critical, it's best to test performance both ways in order to make the best choice.</span></span>

<span data-ttu-id="940c6-143">일반적으로 지연 오프 로드를 설정한 경우에 명시적 로드를 사용할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-143">Typically you'd use explicit loading only when you've turned lazy loading off.</span></span> <span data-ttu-id="940c6-144">Serialization 중 지연 오프 로드 설정 해야 하는 경우 시나리오 중 하나 이며</span><span class="sxs-lookup"><span data-stu-id="940c6-144">One scenario when you should turn lazy loading off is during serialization.</span></span> <span data-ttu-id="940c6-145">지연 로드 하 고 직렬화도 혼합 하지 마세요 및 지연 하는 경우 의도 한 것 보다 훨씬 더 많은 데이터를 쿼리 하 게 주의 하지 않으면 로드가 사용 됩니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-145">Lazy loading and serialization don't mix well, and if you aren't careful you can end up querying significantly more data than you intended when lazy loading is enabled.</span></span> <span data-ttu-id="940c6-146">직렬화 된 형식의 인스턴스에서 각 속성에 액세스 하 여 일반적으로 작동 합니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-146">Serialization generally works by accessing each property on an instance of a type.</span></span> <span data-ttu-id="940c6-147">속성 액세스는 지연 로드를 트리거하고 지연 로드 된 엔터티 serialize 됩니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-147">Property access triggers lazy loading, and those lazy loaded entities are serialized.</span></span> <span data-ttu-id="940c6-148">Serialization 프로세스에는 다음 지연 로드 하는 엔터티를 잠재적으로 더 많은 지연 로드 및 serialization을 일으키는의 각 속성에 액세스 합니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-148">The serialization process then accesses each property of the lazy-loaded entities, potentially causing even more lazy loading and serialization.</span></span> <span data-ttu-id="940c6-149">런어웨이 연쇄 반응을이 방지 하려면 지연 엔터티를 serialize 하기 전에 오프 로드를 설정 합니다.</span><span class="sxs-lookup"><span data-stu-id="940c6-149">To prevent this run-away chain reaction, turn lazy loading off before you serialize an entity.</span></span>
